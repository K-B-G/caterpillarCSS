Caterpillar CSS:What is it?

Caterpillar CSS was created in order to streamline and speed up the process of laying out web pages. It was inspired by systems like Responsive Grid System and Skeleton CSS. Instead of being a framework, it works more like an extensible boiler plate. The system is grid-less, and instead uses a row and block system. You can still design Layouts with a grid since it is implied, but the block layout gives it much more flexibility versus a rigid grid system; which I believe does not fit the web medium as well.

Features

Easy to understand syntax

All classes and id sets are designed to be intuitive, literal, and human readable to make development easier.

Easy Responsiveness Built In

There are only 2 views to design for at the start: Portrait and Landscape. Minimal adjustment and configuration is needed to make a site responsive and well designed; basically almost everything becomes one column in portrait mode.

High Cross-Browser Compatibility

Caterpillar CSS works in IE8+* and most other major browsers. That means less time worrying about compatibility and more time designing. *IE8 and some others still have a few quirks about them, but anyone dealing with IE knows it has to be hacked to work either way. Also for it to work in IE8 that I know of you have to get rid of the media queries.

Extendable and Customizable Boilerplate

Most html elements and pseudo selectors have default styles and attributes that can be changed for style purposes. You can also create new elements and classes to fit your needs like say, different sized blocks.

Light Weight

Caterpillar CSS was designed to not need excess elements to layout a page. Most sizing and styling's can be applied directly on elements. No more divs in divs, inside more divs; seriously why is that I thing.

Quick and Stackable Styling

You can apply many single styles to any element(see the notes, bugs, and more, box below). This means you can code looks on the fly like how you can with inline styles, minus the heresy and head ach later on. Can also work with some component styles.

Caterpillar Workflow

Prototype Phase

Pages can be quickly laid out buy follow best practices like following page flow, and intermediate understanding of css attributes settings. The simplicity of the system coupled with emphasize on human readability makes it easy to prototype sites with.

Feedback Phase

Once prototyped pages can be worked and tested on to check for bugs and issues. Can also be used to see what optimizations can be done later.

Finalization

One layout is finalized refactoring should be a simple process to speed up pages by understanding how the browser parses CSS and HTML. Hmm... bloom filtering.

Rows

Rows are designed to hold different sections of content or "blocks", and to help divided up content on the page evenly. They essentially act as the "bounds" for blocks. The standard row size is 1366em, which will leave enough white space on either side. The other row available as a default is a full width row that spans 100% of the viewports width. Best practices would to either make the <section> or <div> tags as rows. Rows have no stylistic attributes, so they are easy to customize.

Blocks

Blocks of the system are, well... the building blocks. You use these as cards or containers to further divided up and layout content. Blocks range in size from the full width of the "row" down to 7em; in the default boiler plate file. Almost every block has 2em margin on the sides to space them out. Different size blocks can be used together to get desired effects and grid layouts. You can also create new blocks if you need. Blocks are display as inline-block. So when you add let's say, divs together in a "row" when a block is place that exceeds the "rows bounds" it simply breaks below to a new line, and is centered. Blocks have minimum attributes, so they can be heavily styled and customized.

Responsive Layout

There are only two views: Landscape and Portrait. All the standard blocks in landscape views change to display block and essentially become one column when the viewport is in portrait mode. This is where following best practice flow pattern is important; if done right all the blocks and information will be displayed in correct order with almost no adjustments necessary. One block, different views. For landscape views, the media queries were designed to adjust the base pixel font size to scale all the content to fit various screen resolutions; it basically scales the content. Since the only thing changing between screen resolutions are a few property attributes, this helps keep the style sheets smaller and lighter. If let's say you need to have a certain property on an element in landscape, but not in portrait, simple add the property at the bottom of the global landscape media query and have the last few characters be "_l", e.g. ".font_color_purple_l".

ROW

Code

  @media all and (orientation: landscape) {
    .block_6 {
	 box-sizing: inherit;
    display: inline-block;
        vertical-align: top;
        width: 18.3em;    
        margin-right: 2em;
        margin-bottom: 1.6em;
        margin-left: 2em;
        }
        
            .font_color_purple_l {
	 color: #6207B9;
        }
    }

Images

Images are designed to be easily responsive and customized behavior wise. Default is that they display as inline-block, only have a little margin on the bottom, and inherit their height. Images will display to their actual set width by default. If you want them to fill up the container that they are in and be more responsive, simply add the ".full_width" class to them.

Font and Text Controls

The wrapper is what sets most of the global defaults for the whole system. It takes up 100% of the browsers viewport, is displayed as a block, is centered, centers all text and blocks, sets the default font-size of 10px, sets the default global font and color, and background color. Changing the font in the wrapper changes the all the font on all the elements; unless those elements have another font overridden.

ROW

Code

  #wrapper {
    width: 100%;
    height: inherit;
    margin-left: auto;
    margin-right: auto;
    display: block;
    font-family: verdana,
                     sans-serif;
    color: #000000;
    text-align: center;
    background-color: #ffffff;
    }
    
   h1 {
    display: inline-block;
    margin-bottom: 1em;
    margin-left: 0em;
    text-align: left;
    font-size: 4.7em;
    font-family: inherit;
    font-weight: 700;
    }

Customize Styles on the Fly

One of the benefits of this system is that since there are not many attributes set globally but the bare minimum, it is easy to add and customize elements on the fly when need. The overall principle of this is similar to inline css styles, minus the heresy. There are already many default helper css classes available to help in common things like font-color changes, text alignment, padding adjustments, margin adjustments, ect. These should be used conservatively though, in order to maintain simplicity and keep the code from turning into spaghetti.

Another thing about this system is that you can create components out of themes and styles that you use often as demonstrated in the demo below. Remember to always use clear descriptive component names.

ROW

Code

  < div class="block_66 color_white">
   
   .block_66 {
        box-sizing: inherit;
        display: inline-block;
        vertical-align: top;
        width: 85.4em;
        margin-left: 2em;
        margin-right: 2em;
    }
   
    .color_white {
	background-color: #ffffff;
     }

Final Thoughts

Notes

As of now the base typical setting for a row class HTML element like a div is set to 1366em. That size was picked because it's one of the more popular screen sizes as said by this source. However, I think I need to change it to something that's better suited to scaling.

The system is also designed to be flexible and extendable and that you can add other block settings or element settings as seen fit. Majority of the html elements have default style settings that can be altered; thought I still need to add a few. I have set all of the text tags with default spacing and sizing with readability and UI/UX in mind.

Current Bugs

  • Viewport super zooms out when input forms are selected due to android keyboard affecting the viewport ratio; I really, really, hate this. Tried hack methods to get around it, but to no avail... yet. Currently would need to do a jquery/js to adjust the zoom to fix it.
  • Need to fully document the select tag, since I know it has issue with being displayed as border box.
  • Some of the styling's for input elements don't look right in certain browsers; chrome is the only one that is fine... shocker. Need to fix them so there is some browser uniformity.
  • Need to document some common quirks IE8+ has.
  • The class "block_2_column" which is only to make a block display in two block rows will lose effect if the person rotates their phone from portrait, to landscape, then back to portrait. Not sure why this is happening.
  • Sometimes certain font properties like line height won't display the first time a page is rendered; not sure why.
  • Certain blocks will break when phone or tablet is tilted from portrait to landscape. Need to fix this.
  • If you set a monospace font as the global defaults the block system breaks; going to fix this.
